# Review da Plataforma — 29/04/2026

Revisão completa da aplicação `app/` (SPA Vite + React + Supabase) com foco em
segurança, responsividade mobile, fluxos CRUD (cadastro/edição/exclusão de
máquinas), bugs e melhorias de UX/negócio.

> Os bugs marcados como **[CORRIGIDO]** foram efetivamente alterados nesta
> branch (`claude/exciting-clarke-dp7Nl`). Os demais ficam como pendências
> mapeadas para próximas iterações.

## 1. Bugs corrigidos nesta branch

### 1.1 Crash na busca de prestadores quando `brand`/`model` é nulo
- **Arquivo**: `app/src/pages/Machines.tsx:148-157`
- **Sintoma**: `m.brand.toLowerCase()` lançava `TypeError: Cannot read
  properties of null` quando uma máquina cadastrada não possuía marca/modelo
  preenchidos. Bastava 1 máquina com campo nulo para a página inteira de
  `/prestadores` quebrar.
- **Fix**: filtro agora monta um `haystack` defensivo com `filter(Boolean)`
  antes de aplicar `toLowerCase()`.

### 1.2 `operator_type` desalinhado entre formulário e detalhe
- **Arquivos**: `MachineFormFields.tsx`, `MachineDetails.tsx`,
  `AddMachine.tsx`
- **Sintoma**: o select cadastrava `"employee"`, mas a tela de detalhe
  comparava com `"hired"`. Resultado: máquinas com operador contratado eram
  exibidas como "Operador Próprio", ou seja, **mensagem oposta da realidade**
  para o produtor.
- **Fix**: a tela de detalhe agora aceita ambos (`employee` e o legado
  `hired`).

### 1.3 Proprietário podia contratar a própria máquina
- **Arquivo**: `app/src/pages/MachineDetails.tsx`
- **Sintoma**: nenhum guard impedia que o `owner_id` fosse igual ao
  `renter_id`. Isso poluía métricas (transações fictícias) e podia ser usado
  para inflar reputação artificialmente.
- **Fix**: handler `handleBooking` bloqueia + botão da `MachineBookingForm`
  exibe "É sua máquina" e fica desabilitado quando `userId === owner_id`.

### 1.4 Reservas com data no passado e quantidade < 1
- **Arquivo**: `MachineDetails.tsx`, `MachineBookingForm.tsx`
- **Sintoma**: `start_date` aceitava qualquer data; `quantity` usava
  `parseInt(...) || 1` mas não impedia `0`/`NaN` chegando no DB.
- **Fix**: validação client-side + `min={today}` no `<input type="date">`.

### 1.5 Vazamento de subscription do Realtime no Chat
- **Arquivo**: `app/src/pages/Chat.tsx`
- **Sintoma**: `useEffect` chamava `initChat()` (async), e o cleanup retornado
  pela `Promise` era descartado. Resultado: cada navegação abria um novo
  canal `chat-...` sem fechar o anterior — vazamento de memória + duplicação
  de mensagens recebidas.
- **Fix**: `useEffect` armazena `cleanup` da Promise e chama no unmount.
  Também adicionei dedupe por `id` na inserção otimista para casos onde o
  evento `INSERT` retorna pelo realtime depois do otimismo.

### 1.6 Rota `/test-connection` exposta em produção
- **Arquivo**: `app/src/App.tsx`
- **Sintoma**: rota de debug aparecia também em build de produção, expondo
  detalhes de conexão Supabase para qualquer visitante.
- **Fix**: rota envolvida em `import.meta.env.DEV`.

### 1.7 `Documents.tsx` usava `window.confirm()` para excluir KYC
- **Arquivo**: `app/src/pages/Documents.tsx`
- **Sintoma**: UX inconsistente com o resto do app (que usa `AlertDialog`),
  além de bloquear a thread.
- **Fix**: substituído por `AlertDialog` com confirmação explícita.

### 1.8 Tabela admin não scrollava no mobile + page sem reset
- **Arquivos**: `AdminUsersTab.tsx`, `AdminMachinesTab.tsx`
- **Sintoma**: tabelas com 7-8 colunas estouravam o viewport no mobile;
  busca com `w-[300px]` cortava o input em telas <320px; ao trocar filtro
  o usuário continuava na página N (vazia).
- **Fix**: wrapper `overflow-x-auto`, busca com largura responsiva, e
  `useEffect` que reseta `currentPage` para 1 quando filtros mudam.

### 1.9 Bottom-nav: tap target e safe-area iOS
- **Arquivo**: `app/src/components/ui/bottom-navigation.tsx`
- **Fix**: `paddingBottom: env(safe-area-inset-bottom)`, `min-w-[44px]` por
  item, label de `10px` -> `11px` e atributos ARIA (`aria-current`,
  `aria-label` na nav).

### 1.10 Console statements vazando para produção (parcial)
- **Arquivo**: `Chat.tsx` agora gateia logs com `import.meta.env.DEV`.
- Demais ocorrências (32 nos `pages/`/`components/`) ficaram listadas em
  §3 como dívida.

### 1.11 Campo `description` ausente do cadastro
- **Arquivos**: `MachineFormFields.tsx`, `AddMachine.tsx`,
  `lib/schemas/machineSchema.ts`
- **Sintoma**: `MachineDetails.tsx` renderiza `machine.description` e o JSON-LD
  o usa, mas o formulário **nunca capturava esse campo** — toda máquina
  cadastrada nascia com `description = null`. SEO + UX prejudicados.
- **Fix**: campo `<Textarea>` adicionado, contador de caracteres, schema Zod
  com `max(2000)`, persistência em `INSERT`/`UPDATE`.

## 2. Pendências de segurança (alto impacto, fora do escopo deste PR)

### 2.1 Bucket `user-documents` é PÚBLICO
- **Crítico.** `Documents.tsx:108-120` faz upload + insere `document_url`.
  Se o bucket estiver marcado como `public` no Supabase, qualquer pessoa com
  o caminho consegue baixar CPF/RG/CNH/comprovantes.
- **Recomendação**:
  1. Marcar bucket `user-documents` como `private` no painel.
  2. Trocar downloads por `supabase.storage.from('user-documents')
     .createSignedUrl(path, 60)` em `handleDownload`.
  3. Adicionar policy de leitura no Storage que permita apenas o `auth.uid()`
     dono do arquivo + `is_admin()`.

### 2.2 Verificar policies de UPDATE/DELETE em `machines`
- O fluxo de `MyMachines.handleDelete` e `AddMachine` (edit) só dispara
  `.eq('id', machine.id)`. **Sem RLS adequado, qualquer usuário autenticado
  poderia editar/excluir qualquer máquina.**
- **Recomendação**: garantir migration:
  ```sql
  CREATE POLICY "owner manages own machines" ON machines
    FOR ALL TO authenticated
    USING (owner_id = auth.uid())
    WITH CHECK (owner_id = auth.uid());
  ```
  Validar com testes manuais (login B tentando editar máquina do A).

### 2.3 Avatar upload sem validação MIME real
- `Profile.tsx:48-58` (não corrigido aqui) usa `Math.random()` no nome do
  arquivo (colisão possível) e confia 100% no `accept` do input. Recomenda-se
  validar `file.type` + tamanho server-side via Edge Function ou trigger.

### 2.4 Política `user_profiles` admin update muito ampla
- Admin policy permite UPDATE em qualquer coluna. Se um admin for
  comprometido, pode alterar `cpf_cnpj`/`auth_user_id` arbitrariamente.
- **Recomendação**: usar a RPC `admin_set_user_verified` (já existe) e
  remover UPDATE direto, ou restringir via trigger às colunas
  `verified, status`.

### 2.5 Console statements em produção
- 32 ocorrências de `console.error/log` sem `import.meta.env.DEV`. Isso
  vaza payloads de erro para devtools (incluindo possivelmente PII em
  `BookingRequestsList:150`, `KYCUpload:114`).
- **Recomendação**: criar helper `lib/log.ts` único (`devLog`/`devError`) e
  rodar codemod.

### 2.6 Rate limit no client (login/forgot)
- `ForgotPassword.tsx:30` apenas registra um TODO. Recomenda-se um throttle
  client-side (ex. desabilitar botão por 30s após cada tentativa) além do
  rate-limit do Supabase Auth.

## 3. Pendências de UX/Mobile (priorização)

### 3.1 Alta prioridade
1. **MyMachines** — botões `size="sm"` no rodapé do card têm ~32px de
   altura, abaixo do mínimo Apple HIG (44px). Trocar por `size="default"` em
   mobile (`size-[44px]`) ou usar dropdown de ações.
2. **Header sem `safe-area-inset-top`** — em iPhone X+ a notch sobrepõe o
   logo. Aplicar `paddingTop: env(safe-area-inset-top)` no `<header>`.
3. **Card sticky de booking** (`MachineBookingForm`) tem altura ~450px e
   `sticky top-24`, fica inacessível em phones pequenos quando aberto. No
   mobile substituir por **bottom sheet** (Drawer já existe em
   `components/ui/drawer.tsx`) com handle e CTA fixo.
4. **Skeletons em `Search.tsx`/`Dashboard.tsx`** — mostrar bones do
   `MachineGrid` durante o carregamento inicial (já existe `Skeleton`).

### 3.2 Média prioridade
5. **Empty state da home `Machines.tsx`** quando o banco está realmente
   vazio (não filtros) — hoje cai no demo, mas o banner indica isso. Seguir
   exibindo a CTA "Cadastre a primeira máquina".
6. **Confirmação de envio de documento** — após upload com sucesso, atualizar
   status para "Aguardando análise" com data prevista (ex. "Análise em até
   24h").
7. **Notificações** — `Alerts.tsx` tem listagem mas sem sino no `Header`. Um
   badge com contagem de não-lidos aumenta o engajamento.
8. **Reservas — separar `start_date`/`end_date`** na UI. Hoje
   `MachineDetails:203` força `end_date = start_date` (booking de 1 dia
   apenas). Para serviços por hectare/hora, talvez aceitar **período**.

### 3.3 Polimento
9. **Bottom-nav** — destino "Mensagens" aponta para `/dashboard` em vez de
   uma rota dedicada `/dashboard/mensagens`. Considerar criar a página.
10. **Profile** — debounce no botão "Salvar" (evita double-submit).

## 4. Sugestões de melhorias de negócio

### 4.1 Monetização
- **Plano de destaque (anúncio premium)** — para o owner aparecer no topo de
  resultados em determinada categoria/cidade. Receita previsível, baixo
  custo técnico (`is_featured` + ordenação por boost).
- **Cobrança opcional de taxa** — hoje toda a comunicação é P2P (pagamento
  direto). Para começar a monetizar sem afastar usuários, **cobrar apenas a
  partir da N-ésima locação concluída**, ou cobrar o renter (não o owner)
  como "taxa de conexão" pequena (R$ 5-15) em troca de garantias mínimas
  (verificação de identidade do owner, checklist pré-uso).
- **Seguro embutido** — parceria com seguradora para oferecer cobertura por
  contratação (avulsa, ~3-5% do valor). Selo "com seguro" aumenta confiança.
- **Marketplace de implementos/peças** — owner que cadastra um trator
  geralmente também aluga grades, plantadeiras. Cross-sell natural.

### 4.2 Confiança e GTM
- **Selo Verificado mais visível** — hoje aparece no detalhe e no card, mas
  o critério ("documentos analisados") não é explicado em nenhum lugar.
  Criar uma página `/confianca-e-seguranca` listando o processo, e exibir
  ícone de "verificação em andamento" para reduzir frustração.
- **NPS pós-serviço** — `ReviewBooking.tsx` já coleta avaliação. Adicionar
  pergunta NPS 0-10 e usar respostas <=6 como gatilho para um e-mail de
  retenção.
- **Campanha de embaixadores** — produtor que indica e fecha 3 serviços
  ganha 30 dias de destaque. Reuso do destaque já proposto.

### 4.3 Operacional
- **Dashboard owner** com métricas claras: pedidos pendentes, taxa de
  resposta (em < 1h aumenta a visibilidade), tempo médio de resposta.
- **Webhook/notificação por WhatsApp** quando chegar um pedido — o público-
  alvo (produtor rural) raramente está logado o tempo todo. Twilio +
  template aprovado pelo Meta.
- **Calendário de disponibilidade** por máquina (já existe
  `available_from`/`available_until`, mas o cadastro não usa).
- **Mapa em `/prestadores`** — pin por máquina disponível, filtro por raio
  alavanca SEO ("trator em Sorriso/MT").

### 4.4 Crescimento (SEO / programmatic)
- Já está parcialmente listado em `CLAUDE.md` (#3 da dívida). Reforço aqui
  porque o ROI é alto: gerar páginas estáticas em `site/` por
  **categoria × estado** (ex. `/colheitadeira-mato-grosso`) com 10 prestadores
  em destaque + JSON-LD de `ItemList`. Cada página estática indexada vira
  funil orgânico.

## 5. Como testar manualmente o fluxo corrigido

1. **Cadastro de máquina (golden path)**
   - Login → `/dashboard` → "Cadastrar máquina" (`/add-machine`)
   - Preencher: nome, categoria, marca, **descrição**, ano, ao menos 1 preço,
     cidade/estado, raio.
   - Selecionar tipo de operador "Funcionário".
   - Adicionar 1-3 fotos. Confirmar limite (3) e tipos aceitos.
   - Salvar → toast de sucesso → redireciona para `/minhas-maquinas`.
   - Acessar `/prestador/<id>` → verificar que **descrição aparece** e que o
     bloco "Operador Contratado" mostra o texto correto.

2. **Edição**
   - `/minhas-maquinas` → Editar → modificar preço → salvar.
   - Conferir que apenas você (owner) consegue acessar via outro login (deve
     bloquear pelo RLS, ver §2.2).

3. **Exclusão**
   - `/minhas-maquinas` → ícone Lixeira → AlertDialog confirma → toast.

4. **Reserva (negativo)**
   - Logar como dono e abrir `/prestador/<sua-própria-máquina>`. Botão deve
     ficar desabilitado com texto "É sua máquina".
   - Logar como outro usuário, escolher data no passado: validação client-
     side bloqueia.

5. **Documentos**
   - `/dashboard/documentos` → enviar PDF → ver na lista.
   - Clicar Lixeira → AlertDialog (não mais `confirm` nativo) → confirmar →
     desaparece.

6. **Mobile**
   - DevTools → iPhone 14 Pro.
   - Bottom-nav: items com 44px+, labels legíveis, safe-area respeitado.
   - `/prestador/<id>`: specs em coluna única em <640px.
   - Admin (com login admin) → tabelas com scroll horizontal, busca
     ocupando largura total.
